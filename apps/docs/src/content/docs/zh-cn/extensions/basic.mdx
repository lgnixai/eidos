## 1. 摘要

本文档定义了 Eidos 扩展系统的基础架构和公共接口，为可重用、模块化的代码执行环境提供统一的技术规范。扩展系统分为 Script 扩展（数据处理）和 Block 扩展（用户界面）两大类型。

**扩展设计的核心目的**是扩充和覆盖 Eidos 的已有功能，提供轻量级的扩展和个性化定制能力。

## 2. 应用场景与优势

Eidos 的扩展系统将 Eidos 转变为**可塑性软件**，核心理念是**满足个人需求的定制化开发**：

### 2.1 开发理念

1. **解决个人痛点**: 专注于自己遇到的具体问题，比如自定义数据处理流程、界面定制等
2. **快速实验**: 利用 Eidos 的实时预览快速尝试想法
3. **渐进式完善**: 从简单的个人需求开始，逐步完善功能
4. **自主掌控**: 完全控制自己的数据和工具，不依赖外部服务
5. **灵活移植**: 既可以将扩展导出为独立应用，也可以分享到社区供他人使用

通过这种方式，Eidos 扩展系统成为了一个强大的**个人定制化工具平台**，让每个用户都能根据自己的需求打造专属的数字化工作环境，并能够在个人使用与社区分享之间自由选择。

## 3. 术语定义

- **扩展 (Extension)**: 一个可重用的代码单元，封装特定功能
- **Script 扩展**: 负责数据处理逻辑的扩展类型
- **Block 扩展**: 负责用户界面渲染的扩展类型
- **Meta 配置**: 扩展的元数据，定义扩展的入口、类型和行为
- **绑定 (Binding)**: 扩展与其他资源的连接配置
- **Slug**: 用户定义的唯一标识符，用于跨扩展引用

## 4. 核心接口定义

### 4.1 基础扩展接口

```typescript
interface IExtension {
  id: string // 系统生成的唯一标识符
  slug: string // 用户定义的唯一标识符
  name: string // 扩展显示名称
  type: "script" | "block" // 扩展类型
  description: string // 扩展描述
  version: string // 版本号
  code: string // 源代码
  meta?: ExtensionMeta // 元数据配置
  icon?: string // 图标 (data URI)
  marketplace_id?: string // 市场标识符
  ts_code?: string // TypeScript 代码
  enabled?: boolean // 启用状态
  bindings?: Record<
    string,
    {
      // 资源绑定
      type: BindingType
      value: string
    }
  >
}
```

### 4.2 扩展类型概览

- **Script 扩展**: 包含 Tool（工具函数）、TableAction（表格操作）、UDF（用户定义函数）等子类型
- **Block 扩展**: 包含 TableView（表格视图）、ExtNode（扩展节点）等子类型

具体的扩展类型定义和实现细节请参考：

- [Script 扩展规范](./script)
- [Block 扩展规范](./block)

### 4.3 绑定类型

```typescript
type BindingType = "table" | "secret" | "text"
```

## 5. 模块系统

### 5.1 存储架构

扩展代码存储在 SQLite 数据库中，每个扩展通过系统 ID 和唯一的 slug 进行标识。这种设计消除了传统文件系统的依赖，同时使得代码重用和全局引用成为可能。

扩展系统采用平铺式的模块组织结构，取消了文件目录的概念。每个扩展都是一个独立的模块，可以类比为 npm 这个 namespace 下的包——每个包都是唯一的，在文件级别上是平铺的。扩展通过唯一的 ID 进行引用，而不是通过文件路径。

这种设计带来的优势：

- **简化引用**: 无需考虑复杂的文件路径结构
- **全局唯一**: 每个扩展在整个系统中都有唯一标识
- **轻量管理**: 减少了目录结构的管理复杂度
- **便于分发**: 每个扩展都是完整的、独立的代码单元

#### 数据库存储结构示例

以下是三个具体扩展在 SQLite 数据库中的存储示例，展示了 Script 扩展之间的引用以及 Block 扩展引用 Script 扩展的完整案例：

**数据库到虚拟目录的转换逻辑：**

1. **分类组织**: 根据 `type` 字段区分 Script 扩展和 Block 扩展
2. **子分类**: 根据 `meta.type` 字段进一步细分：
   - Script 扩展：`tool` 类型的工具函数
   - Block 扩展：`table-view` 类型的表格视图组件
3. **依赖分析**: 解析 `code` 字段中的 `import` 语句，识别扩展间的依赖关系
4. **层级展示**:
   - `utils` 扩展 (无依赖) 作为基础工具扩展
   - `data-processor` 扩展 (依赖 utils) 提供数据处理功能
   - `chart-block` 扩展 (依赖 data-processor 和 utils) 实现数据可视化
5. **跨类型依赖**: Block 扩展可以引用 Script 扩展的功能，实现业务逻辑与界面展示的分离
6. **信息聚合**: 结合 `name`、`version`、`id` 等字段生成完整的扩展信息

**依赖关系图：**

```
utils (Script)
  ↓
data-processor (Script)
  ↓
chart-block (Block)
```

这种虚拟目录结构让用户能够：

- 直观地查看扩展的组织结构和类型区分
- 了解扩展间的依赖关系，特别是 Block 与 Script 扩展的协作
- 快速定位和管理相关扩展
- 理解业务逻辑 (Script) 与界面展示 (Block) 的分离设计
- 底层仍然保持平铺式的存储优势

**设计理念总结：**

这个完整的例子展示了 Eidos 扩展系统的核心设计理念：

1. **职责分离**: Script 扩展专注于数据处理和业务逻辑，Block 扩展专注于用户界面和数据展示
2. **模块化组合**: 通过 `eidos:` 协议实现跨扩展的函数复用，避免代码重复
3. **类型安全**: 所有扩展都使用 TypeScript 编写，提供完整的类型检查
4. **渐进式增强**: 从基础工具 (utils) 到数据处理 (data-processor) 再到界面展示 (chart-block)，每个扩展都在前一个的基础上增加价值
5. **易于维护**: 清晰的依赖关系让扩展的维护和升级变得简单可控

这种设计使得用户可以灵活地组合不同的扩展来满足个性化需求，既保持了系统的轻量级特性，又提供了强大的扩展能力。

### 5.2 模块引用

扩展系统支持两种模块引用方式：内部扩展引用和第三方模块引用。

#### 5.2.1 内部扩展引用

扩展通过 `eidos:` 协议和唯一的 slug 标识符进行引用：

```typescript
// 定义工具扩展 (slug=utils)
export const add = (a: number, b: number) => a + b
export const multiply = (a: number, b: number) => a * b

// 在其他扩展中通过 slug 引用
import { add, multiply } from "eidos:utils"

// 也可以引用其他扩展
import { formatDate } from "eidos:date-helpers"
import { validateEmail } from "eidos:validators"
```

**引用机制特点**：

- 基于扩展的唯一 slug 标识符
- 无需考虑文件路径或目录结构
- 支持选择性导入（destructuring import）
- 在每个 space 中全局可用

#### 5.2.2 第三方模块引用

扩展系统支持从 ESM.sh 引入第三方模块，但有类型限制：

**第三方模块引用限制表格**：

| 扩展类型        | 内部扩展引用 | 第三方模块引用 | 说明                         |
| --------------- | ------------ | -------------- | ---------------------------- |
| **Script 扩展** | ✅ 支持      | ❌ 不支持      | 保持轻量级，专注数据处理逻辑 |
| **Block 扩展**  | ✅ 支持      | ✅ 支持        | 支持 UI 组件库、图表库等     |

Block 扩展的第三方模块使用详情请参考 [Block 扩展规范](./block)。

## 6. 资源绑定规范

### 6.1 绑定类型

- **table**: 绑定到数据表
- **secret**: 绑定到敏感配置
- **text**: 绑定到文本资源

### 6.2 绑定配置

绑定机制允许扩展访问外部资源而不硬编码敏感信息：

```typescript
{
    bindings: {
        "apiKey": {
            type: "secret",
            value: "openai_api_key"
        },
        "targetTable": {
            type: "table",
            value: "users"
        },
        "configText": {
            type: "text",
            value: "welcome_message"
        }
    }
}
```

## 7. 设计原则

1. **轻量级优先**: 扩展应该是轻量级的功能增强，而不是完整的项目实现
2. **功能扩充**: 扩展的目的是扩充和覆盖 Eidos 的已有功能，提供个性化定制
3. **TypeScript 优先**: 所有扩展代码使用 TypeScript 编写，提供类型安全
4. **AI 友好**: 接口设计便于 AI 理解和生成代码
5. **模块化**: 支持跨扩展的代码重用和组合
6. **安全性**: 通过绑定机制避免硬编码敏感信息
7. **可扩展性**: 支持新的扩展类型和元数据配置
8. **单一职责**: 每个扩展专注于特定功能，保持代码简洁
9. **平铺式组织**: 采用平铺式的模块组织，通过唯一 ID 引用，无需文件目录结构
10. **分层依赖**: Script 扩展保持轻量级，Block 扩展支持丰富的第三方库
11. **渐进式增强**: 从内部扩展到第三方模块，提供渐进式的功能增强能力

## 8. 元数据配置

每个扩展可以导出 `meta` 对象来定义其行为：

```typescript
export const meta = {
  type: "udf",
  funcName: "add",
  udf: {
    name: "add",
    deterministic: true,
  },
}
```

## 9. 安全考虑

- **数据隔离**: 扩展只能访问授权的数据
- **沙箱执行**: 扩展在受限环境中运行
- **绑定验证**: 所有资源绑定都经过验证
- **权限控制**: 基于用户角色的访问控制
- **代码审核**: 市场扩展需要经过安全审核
- **第三方模块限制**:
  - Script 扩展禁止引入第三方模块，避免潜在的安全风险
  - Block 扩展的第三方模块限制在 ESM.sh 源，确保包的安全性
  - 第三方模块的权限限制在 UI 渲染范围内
  - 定期审查和更新第三方模块的安全策略
